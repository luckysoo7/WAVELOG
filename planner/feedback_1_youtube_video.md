## 🔴 Feasibility Gate 먼저 — 저작권 리스크

### 현실 진단 (기획자 + 엔지니어 공동)

"수익창출 포기하면 쓸 수 있다"는 **절반만 맞는 말**입니다.

YouTube Content ID 시스템에서 저작권자는 클레임 시 3가지 중 선택:


| 옵션                  | 결과                                   |
| ----------------------- | ---------------------------------------- |
| **차단 (Block)**      | 영상 즉시 삭제. 3회 = 채널 정지        |
| **수익화 (Monetize)** | 영상 살고, 광고 수익이 저작권자에게 감 |
| **추적 (Track)**      | 통계만 수집, 영상 정상 운영            |

**"많은 플리 채널이 잘 운영되고 있다"가 사실인 이유:**
대부분의 음반사/KOMCA는 "차단"보다 "수익화"를 선택함 → 영상이 살아있음.
실제로 수백만 뷰 로파이 채널들이 이 방식으로 수년째 운영.

**추가 리스크 — yt-dlp 문제:**
오디오 파일을 만들려면 YouTube에서 오디오를 추출해야 하는데,
이 자체가 YouTube ToS §5(b)(iv) 위반. 계정 정지 리스크 별개로 존재.

**결론:** 리스크가 없는 게 아니라, *감수할 수 있는 수준이냐*의 판단 문제.
한국 음악 특화라 국내 레이블 정책이 변수.

## Q1. 저작권/법적 리스크 수용 수준

**세 가지 접근법:**

### 옵션 A — 실용주의 (대부분 플리 채널의 방식)

yt-dlp로 오디오 추출 → FFmpeg 합성 → 업로드.
Content ID 클레임 오면 그냥 수익 넘기고 운영. 차단 오면 해당 곡 제거 후 재업로드.

- 장점: 구현 단순, 지금 인프라로 거의 가능
- 단점: ToS 위반 리스크, 한국 레이블 정책에 따라 채널 날아갈 수 있음
- 현실: 이 방식으로 운영하는 채널 수천 개 존재

### 옵션 B — 보수주의 (법적 리스크 최소화)

YouTube Embed만 활용해 재생목록 영상을 만들지 않음.
대신 **"이번 주 선곡 요약" 쇼츠** 형식으로 → 60초 예고편만 제작.

- 예: 곡 제목 카드 20장 슬라이드 + 배경음악 없음 (또는 크리에이티브 커먼즈 bgm)
- 장점: 리스크 없음
- 단점: 사용자 경험 완전히 다름, 차별화 약함

### 옵션 C — 하이브리드 (추천)

yt-dlp 없이, **각 YouTube 영상을 iframe으로 순서대로 재생하는 웹 플레이어**를
화면 녹화(screen capture)하는 방식으로 영상 제작.

- 이렇게 하면 오디오는 YouTube에서 직접 재생되므로 추출 없음
- 다만 녹화 품질, 실시간 처리 문제 있음
- 서버에서 headless Chrome + Puppeteer + FFmpeg capture

추천: **옵션 A** — 실용주의.
이유: 수천 개 채널의 선례가 있고, 차단 클레임이 오더라도 계정 전체 리스크는
3번 스트라이크 이후이므로, 초기에 소규모로 시작해 패턴을 파악하면 됨.
한국 배캠/별밤 팬 타겟이라 초기 구독자 확보가 더 중요.

> 답변: A가 맞겠네.

## Q2. 영상 포맷 — 무엇이 사용자를 최대한 오래 머물게 하나

### 사용자 관점 에이전트 분석

"플리 유튜브"를 어떻게 쓰는가:

- 공부/업무 중 백그라운드 재생 → 화면 안 봄 → 정적 배경 충분
- 가끔 폰 들어서 "지금 뭐야?" 확인 → 현재 곡 표시 중요
- 댓글로 "몇 분에 X 곡 나와요?" 물어봄 → 챕터 타임스탬프 필수

배캠/별밤 특화 사용자 특성:

- 30-50대 주 청취층 → 모바일 유튜브에서 소비
- "배철수가 고른" 브랜딩에 신뢰감 → 구독 유인
- 향수 자극 → 롱폼 체류 시간 유리

### 옵션별 비교


| 포맷                                       | 제작 난이도 | 사용자 경험 | 알고리즘 노출 |
| -------------------------------------------- | ------------- | ------------- | --------------- |
| A. 정적 배경 + 하단 곡명 텍스트            | ★☆☆      | ★★★      | ★★★        |
| B. 앨범아트 슬라이드 (곡마다 전환)         | ★★☆      | ★★★★    | ★★★★      |
| C. 오디오 비주얼라이저 (스펙트럼)          | ★★★      | ★★★      | ★★★        |
| D. 로파이 감성 루프 영상 (비 오는 카페 등) | ★★☆      | ★★★★★  | ★★★★★    |

추천: **옵션 B (앨범아트 슬라이드)**
이유: FFmpeg으로 각 곡 시작 시 앨범 아트 이미지 전환 + 곡명 오버레이.
앨범 아트가 있으면 썸네일도 풍부하고, 사용자가 "지금 뭐" 확인 가능.
챕터 타임스탬프 + 앨범아트 조합이 체류 시간 가장 높음.

> 답변: 일단은 A정도로 가자. B의 매칭률이 너무 낮았어.

## Q3. 곡 선택 알고리즘 — 어떤 기준으로 20곡 뽑나

이미 DB에 `view_count`가 있으니 데이터 기반 가능.

### 옵션별 접근

**A. 순수 조회수 상위 20곡 (지난 7일)**

- 장점: 데이터 기반, 자동화 쉬움
- 단점: 인기곡 편향 → 매주 비슷한 곡 반복될 수 있음

**B. 조회수 + 다양성 필터 (추천)**

- 조회수 상위 풀에서 → 아티스트당 최대 1곡 → 장르 균형 고려
- 국내/해외 비율 유지 (배캠은 해외 많음)
- 이전 주 영상과 겹치는 곡 제외

**C. 편집장 큐레이션 (반자동)**

- 알고리즘이 40곡 후보 뽑고 → 영수가 20곡 최종 선택
- 장점: 퀄리티 보장
- 단점: 매주 수동 작업 필요 → 자동화 취지와 안 맞음

**D. 혼합 (배캠 10곡 + 별밤 10곡)**

- 두 프로그램 균형 → 구독자가 둘 다 팬이면 최적
- 각 프로그램에서 조회수 상위 추출

추천: **옵션 D + B 기준 적용**
배캠 10 + 별밤 10 각각 선택 시 (아티스트 중복 제외, 이전 주 겹침 제외).
이유: 두 프로그램 팬 모두 커버, 완전 자동화 가능.

> 답변: 일단 배캠이랑 별밤을 다르게 운용할거야. 총 1주일에 2개 올라가는거지. 그리고 B 느낌으로.

## Q4. 채널 전략 — 기존 채널 vs 신규 채널

### 현재 상황

YouTube API OAuth 계정이 이미 있음 (재생목록 만드는 계정).
이 계정에 영상 업로드 vs 별도 채널 개설.

**기존 계정 사용:**

- 즉시 시작 가능
- 재생목록 + 영상이 같은 채널 → 일관성

**신규 채널 개설:**

- 리스크 분리 (클레임으로 채널 날아가도 메인 계정 보호)
- 콘텐츠 전용 채널 → 브랜딩 명확

추천: **신규 채널 개설 + 메인 계정 Sub-channel (YouTube Brand Account)**
리스크 분리하면서 OAuth 메인 계정 유지.

> 답변: 기존계정

## Q5. 자동화 파이프라인 아키텍처

### 엔지니어링 관점 분석

**현재 인프라:**

- GitHub Actions (매일 실행 중)
- YouTube Data API v3 OAuth
- Python + FFmpeg (Actions에 apt-get으로 설치 가능)
- SQLite DB (조회수, 선곡 데이터)

**주간 파이프라인 설계안:**

```
[매주 일요일 오후] GitHub Actions 트리거
  ↓
1. DB 쿼리: 지난 7일 배캠+별밤 선곡 중 조회수 상위 + 다양성 필터
  ↓
2. yt-dlp: 선택된 20곡 오디오 다운로드 (~200MB)
  ↓
3. FFmpeg:
   - 오디오 concat (crossfade 2초)
   - 앨범아트 슬라이드 or 정적 배경 합성
   - 챕터 타임스탬프 삽입
   - 1080p MP4 출력 (~800MB)
  ↓
4. YouTube API: 업로드 + 제목/설명/챕터 자동 생성
  ↓
5. Discord 알림
```

**잠재적 문제:**

- Actions runner 용량: Ubuntu-latest 14GB — 1시간 영상(~1GB) 처리 가능
- Actions 실행 시간: FFmpeg 인코딩 ~10-15분, yt-dlp ~20분 → 총 40분 이내
- yt-dlp 실패율: YouTube 서버 측 차단 시 재시도 로직 필요

**대안 인프라 (Actions 대신):**

- Cloudflare Workers Cron (무료, 하지만 컴퓨팅 제한)
- VPS 서버 (월 $5-10, 가장 안정적)
- GitHub Actions (현재 인프라 그대로, 공짜)

추천: **GitHub Actions 유지** — 현재 인프라 재활용, 추가 비용 없음.
실패 시 VPS로 마이그레이션.

## Q7. MVP 범위 — 첫 영상을 언제, 어떻게 내보낼 것인가

**풀 자동화 (목표 상태):**
매주 일요일 자동 업로드. 개입 없음.

**MVP (첫 3개월):**
반자동 — 스크립트가 영상 파일까지 만들어주면, 업로드는 영수가 확인 후 수동.
이유: 저작권 클레임 패턴, 영상 품질 모니터링 필요.

**실험 계획:**
1주차: 테스트 영상 1개 (비공개) 올려서 Content ID 반응 확인 (48시간 대기)
2주차: 클레임 패턴 확인 후 공개 전환
3주차~: 패턴 안정되면 자동화 전환

추천: **MVP 반자동 → 4주 후 풀 자동화**
이유: 저작권 클레임 패턴 파악 없이 완전 자동화하면 채널 리스크.

## 에이전트 토론 요약

### 🎯 기획자

"차별화 포인트가 명확하다. '라디오 DJ가 고른 주간 베스트'는 알고리즘 큐레이션과 다른 신뢰도가 있음. 배캠 팬덤이 있고 재생목록만으로 발견되기 어려운 구조를 영상으로 보완하는 게 맞는 방향."

### 👤 사용자

"챕터 마커 필수. 플리 영상 볼 때 가장 불편한 게 '지금 뭐 나와?' 모르는 것. 앨범 아트 보이면 더 좋고. 제목에 '배캠×별밤'이 들어가야 검색에서 찾음."

### 🤖 자동화 엔지니어

"현재 인프라(GitHub Actions + YouTube API)로 충분히 가능. yt-dlp가 핵심 의존성이고 불안정할 수 있으니 실패 재시도 + 대체 소스 로직 필요. FFmpeg 인코딩이 병목이지만 Actions 6시간 제한 내 OK."

### ⚖️ 법무 시뮬레이션

"yt-dlp 사용은 ToS 위반이지만 민사 소송 대상 아님. Content ID 클레임은 거의 확실히 올 것 — 단 '차단'이냐 '수익화'냐가 변수. 한국 레이블들은 KOMCA 통해 수익화 선택하는 경향. 비공개로 48시간 테스트 후 패턴 확인 후 공개 전환 권장."
