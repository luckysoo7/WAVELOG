## Q1. 앨범아트 데이터를 언제 가져오나?

현재 구조: 크롤러가 선곡표 크롤링 → YouTube 검색 → DB 저장 → Vercel SSG 빌드
MusicBrainz 검색은 별도 단계가 필요함.

**옵션 A — 크롤러 단에서 미리 저장 (빌드 타임 포함)**

- 크롤러가 MusicBrainz 검색 → MBID + CAA 이미지 URL → DB songs 테이블에 저장
- Vercel SSG 빌드 시 이미지 URL이 HTML에 포함 → 빠름, CDN 캐시 가능
- 단점: DB 스키마 변경 필요, 크롤러 복잡도 증가, MusicBrainz 매칭 실패 시 빈칸

**옵션 B — 클라이언트에서 실시간 조회 (런타임)**

- 사용자가 트랙 hover/클릭 시 브라우저에서 MusicBrainz API 직접 호출
- DB 변경 없음, 즉시 시작 가능
- 단점: 1 req/s 제한 → 여러 곡 빠르게 hover 시 쿼터 소진, 느릴 수 있음

**옵션 C — Next.js API Route 프록시 (런타임 + 서버 캐싱)**

- 브라우저 → `/api/album?title=...&artist=...` → 서버에서 MusicBrainz 호출 → 응답
- 서버 메모이제이션/캐싱 가능, 클라이언트에 API 노출 안 됨
- 단점: Vercel 서버리스 함수 추가, 현재 순수 SSG에서 ISR/서버 혼합 구조로 변경

추천: **A (크롤러 저장)** — 이미 YouTube 검색 파이프라인이 있어서 MusicBrainz 검색을 나란히 추가하면 됨. SSG 순수성 유지, 런타임 API 의존 없음.

## Q2. DB 스키마 변경 범위 (Q1에서 A 선택 시)

songs 테이블에 컬럼 추가 필요:

```
mbid TEXT          -- MusicBrainz Recording/Release MBID
album_name TEXT    -- 앨범명
album_art_url TEXT -- Cover Art Archive 이미지 URL (썸네일)
release_year INT   -- 발매 연도
```

기존 데이터(584곡)는 null로 남고, 이후 크롤링부터 채워짐. 과거 에피소드 백필은 별도 스크립트로 가능.

추천: **진행** — 컬럼 추가는 SQLite에서 ALTER TABLE로 무중단 가능. 기존 데이터에 영향 없음.

## Q3. MusicBrainz 매칭 실패 시 fallback

배철수 선곡 중 한국 가요, 클래식, 비주류 곡은 MusicBrainz 커버리지가 낮을 수 있음.
매칭 실패한 곡의 확장 패널에는 무엇을 보여주나?

**옵션 A — 확장 패널 자체를 비활성화** (앨범아트 없으면 hover 효과 없음)
**옵션 B — 앨범아트 없이 텍스트 정보만** (곡명, 아티스트, YouTube 버튼만)
**옵션 C — 플레이스홀더 이미지** (음표 아이콘 등 generic 이미지)

추천: **B** — 정보가 없어도 확장 패널의 YouTube 버튼 진입점 역할은 유지됨. A는 videoId 없는 곡과 헷갈림.

## Q4. 확장 패널 UI 레이아웃

hover/클릭 시 트랙 아래로 확장되는 패널에 무엇이 들어가나?


추가로 넣고 싶은 정보 있나? (장르, 러닝타임, 앨범 수록 순서 등)

## Q5. 모바일 UX — 두 번 탭 문제

현재: 모바일에서 트랙 클릭 → 바로 YouTube 이동
제안: 모바일에서 트랙 클릭 → 패널 확장 → YouTube 버튼 탭 → 이동

이건 **UX 퇴보**일 수 있음. 기존에 1탭으로 되던 게 2탭이 됨.

**옵션 A — 모바일도 동일하게 확장 패널** (앨범아트 + 정보 보여주고 YouTube 버튼)
**옵션 B — 모바일은 현행 유지** (1탭 → 바로 YouTube), PC만 hover 확장
**옵션 C — 모바일에서 확장 패널, 단 YouTube 버튼을 패널 안에 크게 배치** (탭 영역 넓게)

추천: **A** — 앨범아트를 보여주는 게 이 기능의 핵심이고, 모바일에서도 동일 경험. 버튼을 충분히 크게 만들면 UX 퇴보 체감 최소화.
